# Amtsklar

> A privacy-first AI system that helps people understand German administrative documents without exposing more personal information than necessary.

---

# Vision

Government correspondence in Germany is often written in highly formal legal language that is difficult for many people to understand.

Amtsklar is designed to bridge this gap by transforming complex administrative documents into clear, structured explanations while following Privacy by Design principles.

Unlike traditional document AI systems, Amtsklar does not treat Large Language Models as document readers. Instead, deterministic components extract only the minimum information required before any reasoning takes place.

---

# Core Philosophy

Amtsklar follows five engineering principles.

## 1. Privacy by Design

Privacy is not an additional feature.

It is the system architecture.

Every component receives only the information necessary to complete its responsibility.

---

## 2. Data Minimization

The system should never process or expose information that is unnecessary for the current task.

For example:

Instead of sending

- names
- addresses
- signatures
- complete letters

to an LLM,

Amtsklar extracts only structured legal facts such as

- document type
- authority
- deadline
- amount
- legal basis

---

## 3. Deterministic Before Generative

Whenever a problem can be solved deterministically,
it should be.

Examples include

- document classification
- OCR routing
- field extraction
- date parsing
- amount parsing

LLMs are reserved for reasoning, explanation and contextual interpretation.

---

## 4. Structured Facts Before AI

Language models should receive structured information rather than raw documents whenever possible.

Instead of

4 pages of OCR text

the model receives

{
    document_type
    deadline
    amount
    authority
    legal_basis
}

This improves

- privacy
- cost
- speed
- reproducibility

---

## 5. Explainability

Every answer generated by Amtsklar should be traceable.

Users should be able to understand

- where information originated
- which rule extracted it
- which reasoning step produced the final answer

# Pipeline

## Stage 1

Technical preprocessing

Responsible only for technical inspection.

Determines

- PDF or image
- page count
- rotation
- blur
- OCR required
- confidence

No legal understanding occurs here.

---

## Stage 2

Text extraction

If the document contains selectable text

↓

Native extraction

Otherwise

↓

OCR

---

## Stage 3

Field extraction

The system converts raw text into structured information.

Example

{
    document_type
    authority
    amount
    deadline
    legal_basis
}

---

## Stage 4

Sanitization

Personally identifiable information is removed or transformed.

Examples

- remove names
- remove addresses
- hash contribution numbers
- remove IBANs
- remove signatures

---

## Stage 5

Decision Engine

Business rules determine

- required actions
- missing information
- legal deadlines
- applicable workflows

No LLM required.

---

## Stage 6

LLM

The language model receives only sanitized structured information.

Its responsibilities include

- explaining decisions
- simplifying legal language
- answering follow-up questions
- generating user-friendly summaries

---
